TravelSpark

TravelSpark: Flipping the travel booking experience on its head.

The Travel Industry Issue We have recognized that over the past history of travel and transportation, there has been something particularly missing. So far, many of the traveling based websites are focused on creating the best pricing models to attract potential customers. However, the one truth that many of them are missing is making the connection between the place and the people.

The Solution Allowing the traveler to experience their destination before even arriving, is our main priority. We give travelers the chance to choose where they'd want to go and see based on past experiences from other travelers. The best way to encourage travelers to buy tickets, and improve impulse shopping is to show them what they're missing.

How It Works Users can enter their desired city, and they will be showed a plethora of some of the best images from that location with their respective stories from people who have been there before. If you see pictures that you like, you can put it on a watch list. From the options of pictures that you select, we will curate a special package for you to visit these set places. Travel shouldn't be about prices. Travel should be about the people, the stories, the journey!

What I did: I designed the UI as well as implemented the back-end (Flask), drawing pictures ffrom Google and Instagram of different locations that the user inputs in, as well using Jetblue's dataset to calculate the lowest prices and savings of going to the searched location in comparison to Expedia.

Data Provenance Research

What it is: A Data Derivation Graph (DDG) is a mathematical graph that captures the history of a data analysis. The DDG consists of nodes and edges. DDGExplorer is a data provenance tool developed by Barbara Lerner of Mount Holyoke College with Margo Seltzer from Harvard that allows scientists using R to debug more easily. It captures data flows within and between R scripts and visually displays them into a DDG. Users upload a ddg.txt file, which is created by another program called RDataTracker, and DDGExplorer then parses through it and returns all the necessary information to the scientist.

Find out more here

What I did: I implemented queries in R and Java to capture the execution and elapsed time of each function within the R scripts as well as implemented a query that allowed the user to see how many times each function was called, visually disdplaying that in the DDGExplorer interface. Used Java and R.

Palace: Open-sourcing education.

What it is In websites like Coursera and Udacity, the flow of learning is still very much dictated, with a set of suggested prerequisites developed by the people in the company. With Palace, we want to visualize learning flows in an interesting way and give users the opportunity to build their own learning flows.

We visualize these flows using a graph-like interface, where each node is a subtopic. Users start out with "Intro to CS", and can browse through a collection of videos and tutorials from Youtube, Khan Academy, Coursera, Edx, and other online-self leanring sites. They can add, upvote, and downvote these videos under the topic. When they're done with a topic, they will be presented with different edges that lead to more and more specialized topics, catering to their specific interests. Users link two topics together (say AI and machine learning) by drawing an edge between the two nodes. If enough users create this link, the link will become visible to all users.

What I did: Designed and implemented the back-end database using MongoDB to maximize efficiency of joins between users, courses, topics, and subtopics. Users can connect two subtopics together in the front-end using an edge to connect two nodes (say AI and machine learning), so future users will be able to see machine leanring and AI as part of the same learning track. Likewise, they can connect two courses together in a pre-requisite - succession type relationship. I also hooked the back-end up to a simple Angular front-end interface for testing.

Ask Buzzfeed

What it is:

As the world moves online, research in anthropology and sociology is increasingly becoming based on empirical data sets of social media sites. Questions such as, "What do current youth aged 13-21 think about Islamaphobia?" must consider these datasets. While Facebook and Twitter has opened up their dataset with their API, Buzzfeed, the world's most famous entertainment and news site, has not. Their dataset is valuable as their content reflects the values and trends of the Western pop culture.

What I did:

Because Buzzfeed didn't have an official API (even for GET requests), I created my own. I datascraped Buzzfeed and returned JSON of the titles nad URLs of the posts of a search word, and the entire text of a post given a URL. Then, I created a website that implements the Buzzfeed functionality to display what Buzzfeed thinks of a certain word to the user. Users can input a word, and the website will return the sentiment of Buzzfeed's posts relating to the word, as well as most frequent words associated with that input word. I used the Indico API for sentiment analysis and Django for the web framework. For front-end, I display the sentiment using D3.js data visualization and Reactjs. I am currently working on making the back-end faster and more efficient using parallelization before making it look pretty.
